# Favicon 恢复

## 先分类

- 文件夹图标不是网站 favicon。
- 灰色地球可能是页面从未加载、站点无图标或 URL 映射缺失。
- 书签树错位不是图标问题。
- 只有 URL 书签图标缺失才进入本流程。

## 证据

1. 备份 live `Favicons`。
2. 浏览器关闭后复制数据库作离线分析。
3. 统计书签 URL 在 `icon_mapping` 和 `favicon_bitmaps` 中的精确/可归一化覆盖率。
4. 文件 hash 一致不等于浏览器启动后仍保留映射。

历史案例中，目标浏览器虽然有大量 favicon bitmap 和 URL 映射，但对重建后的书签 URL 精确覆盖极低；从源浏览器数据库按 URL/域名映射可理论补齐大部分。浏览器随后可能清理手工映射，因此数据库补丁只是一条尝试性路线，不是持久成功证明。

## 路线

1. 复制源 `Favicons` 或离线补映射默认只用于副本分析，不直接写回 live profile。
2. 只有确认源/目标 schema 与浏览器版本兼容，连同 WAL/SHM/journal 取得一致快照，使用事务保留目标既有映射，并具备已测试脚本、回滚文件和启动后验证时，才允许写回；技能当前未附带该实现，条件不足时标记阻塞。
3. 写回后覆盖率回落时立即停止重复写库。
4. 可靠刷新通常需要目标浏览器实际访问页面，让其自己生成图标。

批量访问是独立外部网络授权。执行前固定：URL 清单或数量上限、并发、单站超时、总时长、重试次数和连续失败停止阈值。报告历史、缓存、登录态、通知或风控副作用。

剩余项按原因分类：站点失效、需要登录、反爬、跳转后不保留原 URL、无 favicon、特殊协议、本地/内网不可达。不要无限重试。
