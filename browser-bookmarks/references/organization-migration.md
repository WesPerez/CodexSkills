# 书签整理与迁移

## 整理闭环

1. 保留书签栏根部独立 URL、固定高频文件夹、刻意重复快捷入口和用户明确不动的路径。
2. 先做确定项：明显错放的学习、工具、云服务、下载、娱乐和开发资料。
3. 完成全局语义复扫，不把“移动第一批”当作完成。
4. 把候选分成 `确定可动`、`需要用户判断`、`保留为快捷入口`、`误判/不应移动`。
5. 复验 URL 身份、完整树、顺序、书签栏根部 URL 和 checksum。

个人高频入口、同一 URL 的归档+快捷双重用途、跨域仓库/教程/AI 工具不做激进去重。

## 变更路径

运行中的 live profile 优先用临时 MV3 扩展调用 `chrome.bookmarks` API：

- 只申请 `bookmarks`，需要观察时才增加最小 `storage/alarms`。
- 使用执行锁，避免 `onInstalled/onStartup` 并发。
- 只操作已计划节点，记录移动、跳过和错误。
- 云端会回弹时保留短期重复校验能力；稳定前不要移除扩展。

涉及 `removeTree`、整根删除或在线清空重建时，必须先有可审查的 dry-run 计划、目标节点清单、完整备份、幂等执行锁、逐阶段结果记录和失败恢复实现，并对该实现做离线 fixture 测试。技能当前不附带经过验证的整树重建扩展；缺少这些条件时标记阻塞，不得仅按文字步骤在线删除。

内部收藏夹页面可能被 Browser Use URL policy 阻止。不要用 raw CDP 或替代浏览器自动化绕过；改用书签 API 扩展或离线兜底。

## 文件兜底

只有浏览器完全关闭、已有备份且 API/导入不适合时，才直接处理 `Bookmarks`：

- 同时判断 `Bookmarks.bak`，避免主文件恢复后被旧 bak 或浏览器最后落盘覆盖。
- 保留节点字段和 URL 身份，不发明无关字段。
- 重算 checksum。Edge 可能有 `workspaces_v2`，按实际根节点顺序哈希。
- 启动并观察同步窗口；直接文件正确不代表云端接受。

对 `javascript:` bookmarklet 精确迁移，不在整理、同步或 favicon 任务中顺手改写、合并或删除。长 bookmarklet 输出只显示长度和 hash，避免日志污染。

## Edge 到 Chrome

先比较 URL 集合、文件夹数、完整树、同级顺序、根节点和书签栏快捷入口。

- 增量目标：只补缺 URL 时，仍要考虑目标路径和重复快捷入口，不能只做集合并集。
- 原样目标：经过上述安全门禁的 API 清空重建通常比文件替换更能表达用户意图；门禁未满足时不得执行。文件替换容易被 Google 云端旧树合并回来。
- Chrome 桌面 API 通常不能写移动设备书签根；Edge 特有根和移动端差异单独报告。
- Edge/Chrome 根显示名称不同不等于子树不同。规范化 `title/name` 和根名后比较。
- Chrome 首次启动可能创建默认书签或改写文件形态；关闭、启动和重启后都要审计。

历史案例中扩展重建把多棵重复树压回单棵目标树，并在多个同步周期内稳定；剩余重复是源树刻意快捷入口。这个结果只能作为机制证据，未来任务仍以自己的 baseline 为准。
