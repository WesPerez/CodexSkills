## 状态与根因模型

Chromium 书签同步不是“某台电脑的 JSON 覆盖服务器”。服务器按同步实体及 GUID 合并创建、更新和删除操作。整树重建会产生大量新 GUID；旧实体的删除没有成功提交时，云端会同时保存新完整树和旧残树，其他设备随后下载两者。

必须区分：

| 状态 | 证据 | 不能推出 |
|---|---|---|
| 本地树干净 | `Bookmarks` 审计匹配 baseline | 云端已干净 |
| `sync.bookmarks=true` | Preferences 字段 | Bookmarks 引擎已加载 |
| 启动后数量稳定 | 短观察窗口无回流 | 第二设备能下载相同树 |
| API 重建成功 | 运行时节点已变更 | 服务器接受所有提交 |
| 隔离设备下载匹配 | 完整树、顺序和重复组匹配 | 未来漏关设备不会再污染 |

典型同步阻塞证据：

```text
Blocked types: Bookmarks
Failed to download types: Bookmarks
Allow fail: Bookmarks
Commit.Bookmarks: InvalidMessage
```

`InvalidMessage` 不表示云端完全不可读。常见事实是云端仍能下发新旧实体，但拒绝部分更新或删除提交。

## 先取证再选择修复层级

1. 审计 live `Bookmarks`、`Bookmarks.bak`、候选权威备份和顶级重复树。
2. 记录当前 profile、`sync.bookmarks`、`sync.keep_everything_synced`、浏览器进程和策略。
3. 找到同步日志中的首次错误时间、失败类型、实体数量和提交结果；不要因为某次整理发生在错误之后就把整理误判为起点。
4. 判断问题层级：
   - 仅本地文件错误：离线还原即可。
   - 本地正确但开关错误：受控切换 Preferences。
   - 开关正确但引擎阻塞：重建异常实体、重新配置 Bookmarks 数据类型并查日志。
   - 隔离 profile 也下载残树，或删除提交持续 `InvalidMessage`：云端实体污染。
5. 每升级一级都保留上一层证据和回滚点。

Edge 新版 `Bookmarks` 可能在文件尾部带 `sync_metadata`，本地 `Sync Data/LevelDB` 还可能保存失败删除墓碑。只清 `Sync Data`、只替换 `Bookmarks` 或只删一个异常节点都可能让另一处元数据把错误状态恢复回来。检查时把活动 `Bookmarks`、`Bookmarks.bak`、`sync_metadata`、Sync Data 和日志放在同一时间线中；不要默认删除这些对象。

## 自动切换 `sync.bookmarks`

### 允许条件

- 用户已授权开关方向及其云端副作用。
- 精确 browser/user-data-dir/profile 已确认。
- 目标浏览器全部进程已退出，包括 `--no-startup-window` 后台进程。
- `sync.keep_everything_synced=false`。为切换单一类型而顺手关闭 keep-all 会改变所有同步类型，禁止隐式执行。
- 没有企业策略控制同步或同步类型。
- 开启前已有权威 baseline；云端未知或污染时，其他设备已经隔离。

### 修改契约

使用 `scripts/edge_bookmark_sync_switch.mjs`：

- `--expect` 必须匹配旧值，拒绝过期判断；相邻锁文件串行化本脚本实例，覆盖前二次进程/hash 检查降低外部写入竞争。
- 只修改根对象中 `sync.bookmarks` 的布尔 token。
- 拒绝重复 JSON 键、畸形 JSON、缺字段和非布尔值。
- 不整体反序列化后重写 Preferences。
- 在同一阶段备份目录为每次切换保存独立的原始 Preferences、before/after hash 和 mutation manifest；同一目录可以先关闭再开启，不覆盖上一切换的回滚点。
- 临时文件写入、flush 后先复查 live Preferences 仍与读取时 hash 一致，再原子替换；复读证明除目标 token 外字节完全相同。
- 不修改 `Secure Preferences`、Sync Data、密码、Cookie、历史或其他同步类型。

Windows 没有由此脚本持有的浏览器级官方 Preferences 锁，二次检查与原子替换之间仍存在极小的外部启动竞争窗口。发现锁文件、浏览器进程或 hash 变化时一律拒绝；不要自动删除归属不明的旧锁文件。

### 修改后验证

配置变更不是完成条件。启动浏览器后验证：

1. `sync.bookmarks` 没有被浏览器恢复。
2. 同步日志出现 Bookmarks configured/stopped 或等价运行时证据。
3. 0/30/60/120 秒审计本地树。
4. 开启时确认没有意外下载；关闭时确认关闭后重启不会继续回流。

实证中曾出现两种相反结果：

- Edge 完全关闭后离线切换 false→true，但启动后 Bookmarks 没有重新进入同步引擎；必须判定“配置已改、引擎未生效”。
- 另一台已登录设备在 Edge 完全关闭、旧活动 `Bookmarks/Bookmarks.bak` 已隔离后，将 false→true 原子写回并启动，成功从云端下载新树。

差别可能来自浏览器版本、Sync Data 状态、活动书签文件、账号状态或启动序列。未来代理必须验证，不得争论哪一种“理论上应该”发生。

## Edge 多设备云端污染闭环

### 1. 全网隔离

所有非权威设备先关闭**收藏夹同步**并备份本地 `Bookmarks/Bookmarks.bak`。密码、历史、设置等其他类型可以保持原状，除非任务另有要求。任何漏关设备都可能在云端清空后重新上传旧树。

无法接触某设备时，让其关机或断网；在它隔离前不要重置云端。

### 2. 建立权威设备

- 关闭收藏夹同步。
- 恢复并验证权威 baseline。
- 做书签备份；准备重置全部 Edge 云同步时，再做完整 `User Data` 冷备份和关键文件 hash。
- 密码数据库、Cookie 等加密数据通常只可在同一 Windows 用户/机器还原。密码 CSV 是可移植明文，需要独立授权、PIN 和加密保存。
- Cookie/网站登录态通常不是 Edge 常规云同步类型，不需要也不能承诺“重新上传 Cookie”。

### 3. 先尝试低层修复

日志明确指向少量现存节点时，可通过书签 API 在同一父节点和顺序位置复制成新节点，再删除旧节点，以生成新同步实体。保持 URL、名称、子树和顺序不变，只改变内部身份。

若还有异常删除记录、Bookmarks 仍被阻塞，或干净隔离 profile 删除单个异常项仍返回 `InvalidMessage`，停止反复定点重建。此时问题已越过本地层。

不要把若干批 Commit `Success` 当成整个 Bookmarks 类型健康。旧删除墓碑可能在后续周期才重试并再次触发 `InvalidMessage`。至少覆盖多个同步周期、重启和隔离下载。

### 4. 重置 Microsoft 云端同步

这是破坏性外部状态，需要明确授权。Edge 的“重置同步”会删除 Microsoft 服务器上的全部 Edge 同步数据类型，不只是收藏夹。

重置前必须：

- 完整冷备份通过文件数、字节数和关键文件 hash 验证。
- 收藏夹同步关闭，权威书签已恢复。
- 非权威设备收藏夹同步关闭。
- 用户理解密码、历史、设置等云端副本会重新建立。

重置完成提示可能写明“将从此设备上传浏览数据”。立即复查收藏夹开关；如果仍关闭，允许其他类型恢复上传。不要仅凭成功弹窗假设收藏夹也保持隔离。

### 5. 单源上传

1. 重置后先保持权威设备书签树不动。
2. 只在权威设备开启收藏夹同步。
3. 观察 0/30/60/120 秒，随后至少一个更长同步周期。
4. 任何旧残树、异常 bookmarklet、顶级重复树或数量跳升都视为失败，立即关闭收藏夹同步并保全日志。

### 6. 隔离下载验收

使用完全隔离、已认证、只开启收藏夹同步的 profile 或第二设备下载云端树。不能复制权威设备的 `Sync Data` 或 `Bookmarks` 冒充隔离验证。

验收必须比较：URL/文件夹数量、重复组、顶级节点、完整树和顺序、书签栏根部 URL。只比较 URL 集合不够。

认证无法自动完成时，状态为 `部分验证-云端未知`。不要读取、导出或伪造账号凭据来绕过认证。

### 7. 逐台恢复

每次只处理一台：

1. 确认收藏夹同步关闭。
2. 备份本地书签并审计独有项。
3. 清理或隔离本地旧活动树；不要把旧树留在同步开启的 profile 中。
4. 自动开启收藏夹同步。
5. 等待下载并与权威 baseline 完整比对。
6. 通过后再处理下一台；失败立即关闭该机收藏夹同步。

没有“整机优先级”能保证权威机覆盖其他设备。优先级来自操作顺序：旧设备先隔离，云端清空后只有权威机先上传，接收端只下载。某台旧设备漏关后重新联网，已验证干净的云端仍可被再次污染。

## Chrome 云端重复闭环

Chrome/Google 云端和 Edge/Microsoft 云端是两套独立系统。不要因为 Edge 已修复就认为 Chrome 已同步。

当目标是“Edge 树成为 Chrome 云端唯一版本”时：

1. 备份 Chrome live 文件并确认 Chrome 登录和书签同步。
2. 只有具备已审查并经 fixture 测试的扩展实现、dry-run 节点计划、完整备份、幂等锁、分阶段记录和失败恢复时，才可在 Chrome 同步在线时执行 `removeTree/create`。技能当前没有附带该实现，条件不足时标记阻塞，不得照文字直接清空。
3. 立即比较 API 树与目标树，观察多个同步周期。数量曾出现由多棵树膨胀、API 重建后恢复到单棵树并稳定的案例；必须以 baseline 判断，不能硬编码数量。
4. Chrome 根名称和 Edge 根名称可不同；比较子树时规范化根名。
5. 云端再次合并旧树时，用同一扩展按 baseline 压制并记录回弹时间；不要一次执行后宣布成功。
6. 文件替换只能修本地。云端开启时，它可能被旧树覆盖或再次合并。

不要默认清除 Google 账号全量同步数据。只想修书签时优先 API 重建。

## 日志证据

优先查目标 profile 下同步诊断文件和浏览器调试日志。版本不同，路径与字段可能变化；先定位近期写入的日志，再搜索：

```text
Bookmarks
Blocked types
Failed to download
Allow fail
InvalidMessage
INVALID_MESSAGE
ProcessGetUpdates
Commit
configure
stopped
```

日志结论必须与 Preferences 和书签树交叉验证。网络请求成功但 Commit 返回 InvalidMessage，说明不是普通断网。

## 观察窗口和停止条件

建议观察点：变更前、启动后立即、30 秒、60 秒、120 秒、重启后。高延迟环境增加更长周期，但不要无界等待。

立即停止并关闭收藏夹同步：

- URL/文件夹数显著跳升。
- 出现第二棵顶级主文件夹或旧残树。
- 大量重复组增加。
- 权威 baseline 中不存在的异常 bookmarklet 回流。
- `InvalidMessage`、Bookmarks blocked 或失败提交持续出现。
- 其他设备尚未隔离。

## 失败模式矩阵

| 现象 | 根因候选 | 下一证据 | 处理 |
|---|---|---|---|
| 直接替换后启动恢复旧结构 | 浏览器缓存、`.bak`、云端回写 | live/bak/同步开关/时间线 | 关闭浏览器，API 变更或完整还原后复验 |
| UI/Preferences 显示开启但不上传 | Bookmarks 引擎阻塞 | 同步日志和实际 datatype | 修异常实体或重新配置；不要反复盲开关 |
| 改 Preferences 后值正确但无同步动作 | 引擎未重新加载 | configured/stopped 日志 | 标记引擎未验证，换受控启动序列或原生事件 |
| 开启后立即下载大量旧树 | 云端仍污染 | 隔离 profile、顶级树、GUID/重复 | 立即关闭，进入云端污染闭环 |
| 删除后重新回来且 Commit InvalidMessage | 服务器拒绝异常实体删除 | 干净隔离 profile 复现 | 停止本地重试，明确授权后重置云端或联系支持 |
| 多批 Commit Success 后错误复发 | 失败墓碑在后续周期重试 | 更长日志窗口、重启、sync metadata | 不宣告完成，继续到隔离下载或云端重置 |
| 云端重置后提示自动上传 | reset 自动恢复同步 | 逐类型 Preferences 和本地树 | 确认收藏夹仍关闭，再单源开启 |
| 第二设备少/多一条 | 本机新书签、移动端根、上传延迟 | URL 差异和完整路径 | 判断预期新增还是未完成同步，不按数量猜 |

## 其他设备交接模板

```text
使用 browser-bookmarks。目标是从已清理的云端接收权威书签，不允许上传本机旧树。
1. 定位当前浏览器活动 profile，确认收藏夹同步关闭。
2. 备份 Bookmarks/Bookmarks.bak，运行 audit 并保留报告。
3. 将本地旧活动书签隔离或按权威 baseline 清理；不要删除其他浏览器数据。
4. 浏览器完全关闭后，使用 edge_bookmark_sync_switch.mjs 原子开启收藏夹同步。
5. 启动浏览器，按 0/30/60/120 秒审计，并在重启后复验。
6. 完整树、顺序、重复组与权威 baseline 一致才完成；异常立即关闭收藏夹同步。
```
